# 美团 AI 搜索智能体（小团）Reward System 构建情况总结报告

> 数据来源：学城文档《Metric Checklist》（[collabpage/2752458757](https://km.sankuai.com/collabpage/2752458757)）及其全部 11 个子文档。

## 一、总体定位与目标

根据《Metric Checklist》主文档，当前这套体系的根本目标是**推动评测能力向训练侧奖赏系统（reward system）迁移**：过去这些评估维度大多只存在于离线评测（EVA）体系中用于事后打分，现在要把它们逐一转化为可以直接参与强化学习训练的 reward function，为美团 AI 搜索智能体"小团"提供训练信号。

主文档用一张"模型能力 × 问题类型"的网格图来标定现状：绿色表示评测与训练奖赏系统均已覆盖的能力，橙色表示评测侧已具备但训练侧尚缺失的能力。基于这张图，文档给出了三条核心结论，这也是理解整个 reward system 建设阶段的关键背景：

第一，**过程指标缺失是当前最大的短板**，尤其是与 core agent（思考、反思、工具调用）相关的大量指标目前只能在评测侧看到结果，尚未转化为可训练的奖赏信号。第二，**多轮奖励机制系统性缺乏**，这被列为高优先级事项，意味着当前 reward system 主要还是围绕单轮场景构建。第三，**Golden 测试集大量缺失**，不过回复相关的 Golden 集已经在建设中，这是保证 reward 打分稳定性和人机一致性的基础设施，目前还处于补齐阶段。

围绕这三条结论，文档确立了一套**责任机制**：每一个 metric 都指定专人负责，确保"数据 → 提示词 → 模型"形成闭环，防止有指标定义了却没有人跟进落地。同时确立了**执行标准**，分三步走：算法同学先制定评估标准和种子 query，与产品同学拉齐标注口径；产品同学据此产出 1000 量级的 Golden 评测集，用来牵引提示词的迭代，并核验人工标注与模型判分之间的一致率；最后才是 metric 准入，正式录入 reward system 参与训练。

从方法论上看，这套体系里绝大多数 metric 采用 **LLM-as-Judge** 的方式实现——即写一段结构化的评估 prompt，让判分模型阅读 agent 的 query、上下文、思考轨迹、工具调用、工具返回或最终回复，输出 JSON 格式的分数（0/1、0~1 连续值或 null）加理由；少数格式类、规则明确的指标（如工具调用格式、参数合法性）则采用 **rule-based 代码脚本**直接判定，不依赖大模型主观评估。

下面按照"评估对象所处的执行阶段"这条主线，把全部 metric 归纳为六大维度：过程类指标（思考与工具提槽）、Skill 路由类指标、工具结果类指标、回复结果类指标、多轮对话类指标、安全合规类指标。这基本对应了一个 agent 从"接收 query → 思考 → 选 Skill → 调工具 → 拿到工具结果 → 反思 → 生成回复"的完整链路。

## 二、过程类指标：思考与工具提槽

这一大类指标评估的是 agent 在"想"和"调用工具"这两个环节上的表现，是主文档中反复强调"大量缺失"的核心区域，因为这类信号通常隐藏在中间过程里，比最终回复更难捕捉，但对训练 agent 的推理能力至关重要。

**首轮意图理解**评估的是 agent 第一次思考（first_think）是否准确识别了用户 query 中的全部客观约束条件，输出 intent_score。这是整条链路的起点：如果第一步理解就出偏差，后面的工具调用和回复都会跟着跑偏。

**thinking query 纠错**关注的是一个更细节的能力：当用户 query 里出现错字（同音误写、近形误写、输入法误触、专有名词误写等）时，agent 的第一段思考是否具备自主纠错能力。评分逻辑是比例型的——score 等于正确纠正的错字数除以错字总数，如果 query 本身没有错字则跳过不评（null）。这里也明确划定了"过度纠错"的负面情况，比如借纠错篡改了用户的硬约束（把"预算100以内"错纠成"100以上"），这种情况会被扣分。

**工具提槽格式规范与客观条件满足度**这部分是一组三合一的评估，同时给出 search_score（搜索词是否准确）、rewrite_score（改写是否合规）、args_match_score（参数是否匹配客观条件）三个维度的分数，用于衡量 agent 把思考结果转化为工具调用参数这一步是否做到位。

**工具提槽幻觉**专门识别工具调用环节的三类幻觉：COT_TO_TOOL_MISMATCH（思考内容与实际工具调用不一致）、UNSUPPORTED_SLOT_INJECTION（工具参数里塞入了思考过程中未曾出现、凭空捏造的槽位信息）、REWRITE_DRIFT（改写过程中偏离了用户原始意图）。这类指标本质上是在防止 agent 出现"说一套做一套"的行为。

**tool call 格式检查**是这一大类里唯一采用纯 rule-based 实现的指标，代码逻辑包含 `check_exclusive_tool_calls`、`_check_load_skill`、`_check_hard_constraints`、`_check_format` 等多个校验函数，逐项检查工具调用格式是否规范，全部通过才给 1.0 分，否则给 0.0，不涉及大模型主观判断，胜在稳定可控。

**虚假工具/参数检测**同样是 rule-based 实现，从 `reward_context.core_agent.tools` 里遍历全部工具调用，基于预先构建的合法 schema 映射逐条校验：工具名是否在合法集合内、参数能否解析为合法 JSON、必填参数是否齐全、是否存在额外非法参数、参数类型是否匹配。输出结果包含 score（合法调用占比）、status（全部合法/部分合法/全部不合法/出错）以及详细的违规明细，是一套相对成熟、逻辑严谨的格式安全网。

**reflection（反思）相关指标**则围绕 agent 在多步工具调用之间的自我修正能力展开，具体包含四类：react 信息不足判别与 reflection 信息冲突判别（判断 agent 是否正确识别出当前信息不足以回答，以及是否发现了信息间的矛盾）；反思幻觉/过程幻觉（识别 cot 中虚构的供给实体、属性或错误归因，并给出具体的 hallucinated_cot_step_ids，方便定位到具体哪一步思考出了问题）；react 增益判断（用有效反思次数除以总反思次数，衡量 agent 的反思是否真的带来了信息增益，而不是无意义的重复思考）；reflection 过时供给识别（检查 agent 能否在反思阶段主动发现工具返回的供给已经过时）。

**工具调用顺序合规性**是一套针对本地生活搜索场景定制的规则型评估，规定了四条硬规则：宽泛探索类/形容词风格类/模糊概念类/知识背后有消费需求类 query 必须先调用内容搜索工具再调供给搜索工具；查询店内商品必须先拿到真实 poiId 再进行店内搜索；沿途/路径类场景必须先定位到具体地标再多次搜索，不能直接把起点或终点当作地理位置参数；找 POI 类意图必须在搜索完成后、回复前调用详情工具补全信息。这套规则本质上是把"工具使用的经验规范"显式化为可自动检查的逻辑。

## 三、Skill 路由类指标

小团支持 8 种业务 Skill：找优惠（coupon-search）、通用搜索（general-search）、商品搜索（item-search）、手艺人（technician-search）、客服转接（customer-service-transfer）、预约（reservation-search）、医药类（medicine-search）、酒店旅游（travel_and_hotel-search）。这一维度下有两个评估指标，分别对应"选没选对"和"选对了以后干得对不对"。

**Skill 路由合理性（skill_score）**判断 agent 针对当前 query 加载的 Skill 是否与用户真实意图匹配。评估流程设计得比较严谨，要求先判断用户意图属于九种类型中的哪一种（找店、找商品、找手艺人、找优惠、预约、医药健康、酒店旅游、客服售后、其他闲聊），再对照一张"意图类型 → 应选 Skill"的映射表确定标准答案，最后与 agent 实际选择的 Skill 做比对。文档特别列出了几个高频易错场景，比如单纯"找个理发店"应该用通用搜索而非手艺人 Skill，没有明确预约表述的找店需求不该触发预约 Skill，宠物医疗和医美护肤不该误入医药类 Skill，这些细节说明这套评估标准是从真实 badcase 中反复打磨出来的。

**Skill 激活后行为合规性（skill_behavior_score）**则是在确认 Skill 选对之后，进一步检查激活该 Skill 后的工具调用序列是否符合这个 Skill 的专项规则。每个 Skill 都有自己的一套硬规则，例如找优惠场景必须先调 `get_coupon` 再调搜索工具且顺序不可颠倒；客服转接场景一旦激活就不允许再调用任何搜索工具，唯一动作是直接回复；手艺人场景必须使用专用的 `technician_work_search`，形容词风格描述必须先经过内容搜索转化为具体款式名词；酒店旅游场景中机票、火车票、旅行规划等必须走 `agent_handoff` 转人工/其他系统，不能用搜索工具硬顶；医药类场景则统一要求全部走 `agent_handoff`，不允许直接调用搜索工具。这套规则等于是把每个业务线的"操作手册"编码成了可自动核查的行为规范。

## 四、工具结果类指标

这一维度评估的对象是工具刚返回的原始供给结果，也就是在 agent 生成最终回复之前，先检验"拿到的素材本身好不好"。

**供给客观条件不相关**区分了两种计分逻辑：对于非精准找商家场景（比如"附近便宜火锅"这类泛检索），采用比例计分，score 等于 1 减去不相关供给的占比；对于精准找商家场景（用户明确指定了店名或品牌），只要返回结果中命中了目标实体就给满分，否则直接判 0 分，这个设计反映出对"精确匹配"和"模糊召回"两类场景采用了不同的容忍策略。

**主观条件弱相关**处理的是相对更难量化的场景，比如用户说"便宜""安静""适合约会"这类主观偏好，评估逻辑是识别有多少条供给"从普遍经验看较难满足"这类偏好（比如用户想要安静但返回的是主打夜场的酒吧），同样采用弱相关占比来反向折算分数，并且明确把这类"弱相关但不冲突"的情况和"完全不相关"（客观条件冲突）区分开来，后者属于更严重的问题。

**供给过时效**检查工具原始返回结果中的时效性问题，比如商品券已过期已下线、商家已歇业已打烊等，需要结合 current_time 与供给自身携带的时间字段综合判断，用过时供给占比反向折算得分。值得注意的是，"回复结果提示词"部分也有一个类似名字的"回复商卡过时效"指标，两者评估对象不同：一个针对工具刚返回的原始结果，一个针对最终呈现给用户的回复内容，这是同一类问题在链路两个不同节点上的双重把关。

## 五、回复结果类指标

这是内容最丰富的一个维度，因为最终回复是用户唯一能直接感知到的产出，也是历史上评测体系最先覆盖、目前 Golden 测试集建设最领先的部分。

**内容、供给相关性、多样性**是一组三合一评估，包含供给相关性（推荐的商家/商品/团购是否匹配用户显性和隐性需求，比如距离是否合理、是否在营业）、内容相关性（文本描述是否直接回答了问题、文本与卡片信息是否一致、有无幻觉）、供给多样性（面对"推荐附近餐厅"这类泛需求时是否提供了足够丰富和差异化的选择，而对于用户明确寻址单一目标的场景则默认给满分，不强求多样性）。这组评估的案例设计得比较细致，专门区分了"距离稍远但仍可达"（不算不相关）和"跨城/极远"（判为不相关）这类边界情况。

**事实性幻觉**是这一维度里最复杂的一个指标，需要验证回复内容与工具结果之间在 12 个闭集维度上的一致性：名称、地址、距离、评分、价格、服务属性、分类、deal 与 spu 归属关系、数值、预定时间、榜单、营业状态。这个指标最值得关注的设计是它内置了一套**宽松豁免原则**——距离误差在 1 公里以内不判错，价格误差在 20% 以内不判错——这种设计明显是为了避免把正常的表述性误差、四舍五入或数据轻微滞后误判为"幻觉"，从而让这个高风险指标的判罚更贴近实际业务容忍度，减少误伤。

**拒答幻觉**评估的是两类相反方向的错误：False Rejection（该回答的却拒绝回答）和 False Negative Claim（明明有供给却声称没有），这类指标关注的是 agent 在"该说不"和"不该说不"之间的平衡能力。

**首段/top-1 供给不相关**单独针对回复中第一段文字和排在第一位的供给做专项检查，因为这是用户最先看到、权重最高的内容，一旦出错影响面更大。

**回复观点/商卡重复**用 non_redundant_claim_ratio 衡量同一次回复内部是否存在大量重复的观点表述或重复的商家卡片。**回复商卡过时效**则是前文提到的、针对最终呈现给用户的商卡内容做的时效性二次校验。

## 六、多轮对话类指标

主文档明确指出多轮奖励机制是当前系统性缺失、优先级最高的建设方向，从子文档内容看，这部分的 prompt 设计已经相对完整，等待录入训练侧。

**多轮意图理解-指代消歧能力**判断当用户使用代词或指示词（"这两家店""那苏州的呢"）时，agent 是否正确识别其指向。**多轮有效硬约束完整性**在指代消歧的基础上，进一步检查工具提槽是否遗漏了应从历史对话中继承的硬约束条件，比如用户话题从杭州平移到苏州时，是否还记得延续"带宠物""预算500以内"这些之前提过的条件。**多轮意图理解-意图整合能力**则区分了三种多轮意图变化模式：问题细化对应硬约束累加、问题泛化或话题转移对应仅需识别当前轮约束、重复提问对应需要结合前轮信息反思用户真实意图，这套分类逻辑体现了对多轮对话动态特性的细致建模。这三个维度共用一套评估 prompt，同时输出 ambiguity_score、cot_score、tool_call_score 三个分数，分别对应"是否正确消歧""思考是否理解意图""搜索工具入参是否满足意图"。

**多轮重复与多轮矛盾**检查的是回复内容层面的一致性问题：矛盾指的是同一实体的客观属性（价格、距离、营业时间）在前后轮描述不一致；重复指的是当前轮推荐的商家或商品中有 3 家及以上与前轮完全相同，且这里也设有豁免条件——如果用户明确要求"把刚才推荐的再发一遍"，这种符合用户指令的重复不算违规。

**前轮意图纠错**评估的是一种更进阶的能力：当用户进行"重复追问"（换种说法、强调某条件、表达不满）时，这通常意味着上一轮回答存在问题，agent 是否能在思考过程中意识到前轮的错误或遗漏并主动纠正，而不是延续错误逻辑继续输出类似的回复。

## 七、安全合规类指标

这一维度覆盖两个阶段：思考过程中的风险预判，以及最终回复的合规兜底。

**合规性-cot 软拒识**评估的是思考轨迹本身对高风险 query 的风险意识，覆盖五类高风险 query：不合规类（违法、欺诈、规避监管等）、不合理/无法完成类（超出能力边界或物理不可能的任务）、攻击性类（辱骂威胁仇恨等）、诱导性类（提示词注入、要求泄露系统规则）、反事实类（把虚假前提当既定事实强迫模型给出执行性结论）。这个指标设计中有一个很值得注意的细节，叫"防线失守检测"：即使 agent 的思考过程表面上提到了风险提醒，但如果后续依然接受了用户虚构的授权前提并规划了实质性的有害执行步骤（比如案例中"表面提醒但依然打算给出勒索病毒代码框架"），也要判定为失败，这种设计能够防止 agent 通过"先免责声明、再违规执行"的方式蒙混过关。

**安全隐私风险-回复质量**则是对最终回复的合规兜底检查，覆盖四大类红线：涉政涉黄等违法内容和道德底线问题、隐私泄露（用户手机号、身份证、住址等敏感信息未脱敏，或主动提及用户年龄职业等画像信息）、生态外导（美团自身有对应业务却引导用户去竞品平台下单）、系统信息与内部逻辑暴露（暴露系统 prompt、工具名称参数，或在纯文字部分直接暴露供给 ID，或暴露内部的 thinking 决策逻辑）。这个指标同样设置了大量豁免条件，比如用户主动搜索"情趣酒店"这类合法业务范围内的需求不算涉黄；商家的公开联系电话不算隐私泄露；商品名称本身包含竞品词（如"京东优选亲肤棉被子"）不算外导；政务或公共服务如果在美团生态内无唯一替代渠道，允许引导至官方渠道（如 12306、社保公积金查询）。这些豁免设计说明团队在制定安全规则时，特别注意避免"一刀切"导致大量正常业务场景被误伤。

## 八、从 agent reward system 角度的总结与解读

如果跳出逐条 metric 的罗列，从"这是一套用来训练智能体的奖励系统"这个视角回看，可以看到几个比较清晰的设计取向和当前所处的建设阶段。

首先，这套 reward system 的整体架构其实是在**复刻 agent 的 ReAct 执行链路**，而不是简单地只看最终结果打分。从首轮意图理解、思考纠错、工具提槽、工具调用格式与顺序，到 Skill 路由与行为合规，再到工具结果质量、反思增益、最终回复质量，几乎每一个中间节点都设计了对应的奖励信号。这种"过程可观测、可监督"的设计思路，本质上是希望训练出的 agent 不仅结果对，而且推理路径本身是可信、可解释、符合规范的，这比单纯用最终答案的好坏去做结果奖励（outcome reward）要精细得多，也更契合当前业界对 agentic RL 训练中"过程奖励"（process reward）价值的普遍认知——单纯的结果奖励容易导致 reward hacking（比如为了拿到好结果而虚构工具调用信息、跳过必要的核验步骤），而过程类指标恰恰能在训练早期就纠正这些倾向。不过，正如主文档自己指出的，这部分恰恰是当前"评测已具备、训练侧缺失"最集中的区域，说明这套体系目前仍处于**从评测框架向可训练奖励信号转化的中期阶段**，很多设计精良的过程类 LLM Judge prompt 还没有真正接入到强化学习的训练循环里产生梯度信号。

其次，**LLM-as-Judge 与 rule-based 两种实现方式的搭配使用是经过刻意权衡的**，而不是随意选择。可以观察到一个规律：凡是涉及语义理解、主观判断、幻觉识别这类需要"读懂内容"的评估，几乎全部采用 LLM Judge（意图理解、幻觉检测、相关性、安全合规等）；而凡是有明确、封闭规则可以枚举穷尽的评估，比如工具调用的 JSON 格式是否合法、参数类型是否匹配 schema、调用顺序是否符合业务规则，则坚决采用 rule-based 脚本。这种搭配的好处很直接：rule-based 部分提供了完全确定、无幻觉风险、几乎零成本重复调用的"地基层"奖励信号，可以放心地作为训练早期的强约束；而 LLM Judge 部分虽然引入了额外的模型依赖和主观性风险，但覆盖了规则本身无法穷尽的复杂语义空间。这种分层恰好也解释了为什么"格式规范""参数合法性"这类指标反而是目前少数已经比较成熟、可直接使用的部分——因为它们不需要依赖 Golden 测试集去校准，天然稳定。

第三，**大量豁免规则的存在，反映出这套系统在"从评测标准走向训练信号"过程中对 reward hacking 和过度惩罚的双重防范意识**。无论是事实性幻觉里的"距离误差1公里、价格误差20%以内不判错"，还是安全合规里的"情趣用品合理供给不算涉黄""商家公开电话不算隐私""竞品词出现在商品名里不算外导"，这些细粒度的例外条款不是可有可无的补充，而恰恰是这类 reward function 能否在真实训练中稳定发挥作用的关键。如果没有这些豁免，一个追求高 reward 的模型很可能被训练出"过度保守"的行为模式——比如干脆不推荐任何可能有轻微数据误差的供给，或者对所有敏感词一律回避正常业务需求，这在实际的强化学习训练中是一种常见且危险的失效模式（reward 设计过严导致策略退化）。反过来看，"合规性-cot软拒识"里的"防线失守检测"设计，则是在防范另一个方向的风险：agent 学会用"表面安全话术+实质违规执行"的方式钻 reward 的空子，这种细节说明设计者对 RLHF/RLAIF 训练中模型投机取巧的模式有比较深的实战认知。

第四，**多轮奖励机制的缺失是当前最值得关注的短板，但也是相对容易补齐的一块**。从子文档内容看，多轮相关的评估 prompt（指代消歧、硬约束继承、意图整合、多轮矛盾与重复、前轮纠错）在设计完整度上并不逊色于单轮指标，说明"缺失"更多指的是尚未进入训练闭环，而不是方法论层面的空白。这类指标的训练价值其实相当高，因为多轮场景下的错误（比如遗忘约束、自相矛盾、屡教不改）往往是用户流失的直接原因，比单轮的小瑕疵更容易造成实际业务损失，建议在推进优先级上应当尽快把这批已经成型的 prompt 转化为可用于训练的 reward signal。

第五，**Golden 测试集的建设节奏，直接决定了这套 reward system 从"能打分"到"打分打得准且稳"的关键跨越**。文档里定义的执行标准——算法定义标准与种子 query、产品标注产出千级 Golden 集、核验人机一致率、最后才准入 reward system——这个流程本身设计得比较严谨，体现出团队清楚地知道 LLM Judge 类的 reward function 最大的风险在于自身的判分不稳定或与人类判断偏离，如果不做这层校准就直接拿去做训练信号，很可能训练出一个只是在"讨好评分模型"而非真正提升用户体验的策略。目前回复相关的 Golden 集已经在建设中，这与回复类指标本身是历史上最成熟、案例最丰富的部分相互印证；而过程类和多轮类指标由于本身还处于评测转训练的过渡期，其 Golden 集建设大概率会滞后于回复类，这也是后续需要重点投入资源的方向。

总的来看，这套 reward system 目前呈现出"结果类指标成熟度高、过程类和多轮类指标设计完备但尚未训练化、格式与安全类指标稳定可靠"的结构性特征，责任到人的机制和标准化的准入流程为后续持续迭代提供了制度保障，接下来的核心攻坚点集中在把已经写好的过程类与多轮类评估 prompt 真正转化为参与模型训练的奖励信号，并同步补齐对应的 Golden 测试集，这也正是主文档三条结论所指向的下一阶段工作重心。
